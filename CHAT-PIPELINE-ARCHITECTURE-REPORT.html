1. Executive Summary

Verdict: The VS Code Chat UI can be fully repurposed for custom CLI-based AI assistants with zero core rendering changes.

The swap point is IChatAgentImplementation.invoke(). Everything above it (UI, orchestration, tools, streaming) is reusable as-is. Everything below it (the LLM backend call) is what we replace with our CLI assistant pipeline.

Our architecture: embed code-oss (served as a web app, one instance per worktree), build a VS Code extension that registers a chat participant, and pipe data from CLI coding assistants (Claude Code, Copilot CLI, Codex, etc.) through the ChatResponseStream API. The existing UI renders everything automatically.

Manages chat sessions. Key methods:

Path B: Runtime API (implementation)

const agent = vscode.chat.createChatParticipant('our.assistant', handler);

Implementation in extHostChatAgents2.ts. Bridges to main thread via MainThreadChatAgents2 (mainThreadChatAgents2.ts:100).

Default agent priority: When multiple extensions claim isDefault: true, the system prefers extension agents over core agents (_preferExtensionAgents() at chatAgents.ts:505). Our extension's agent will take priority over Copilot's.

6. Response Streaming

Extension Host → Main Thread → UI

  1. ChatAgentResponseStream (extHostChatAgents2.ts:44) creates the vscode.ChatResponseStream API object
  2. Stream methods (markdown(), progress(), etc.) call _report() (line 117)
  3. _report() queues progress DTOs into a sendQueue array (line 92)
  4. A microtask batches the queue and sends via RPC: $handleProgressChunk(requestId, sendQueue) (line 105)
  5. MainThreadChatAgents2 receives chunks, looks up the pending progress callback by requestId (line 472)
  6. Calls model.acceptResponseProgress(request, progressItem) (line 1079 of chatServiceImpl.ts)
  7. ChatModel.acceptResponseProgress() (chatModel.ts:2758) updates the ChatResponseModel
  8. Fires onDidChange → ChatListItemRenderer re-renders the appropriate content parts

8. Tools, MCP Servers & Skills

8.1 Language Model Tools

Service: ILanguageModelToolsService (tools/languageModelToolsService.ts:527)

Unified registry for all tools. Three sources:

  • Built-in tools (builtinTools/tools.ts) — EditTool, AskQuestionsTool, ConfirmationTool, RunSubagentTool, etc.
  • MCP tools — registered by McpLanguageModelToolContribution from running MCP servers
  • Extension tools — registered via vscode.lm.registerTool()

Tools are passed to the agent in IChatAgentRequest.userSelectedTools. The tool call lifecycle:

LLM decides to call a tool
  → response.beginToolInvocation(toolCallId, toolName)
  → UI shows tool progress part (spinner)
  → toolsService.invokeTool() executes it
  → Confirmation flow if needed (UI shows approve/reject)
  → Result sent back to extension → LLM continues

8.2 MCP Server Architecture

Location: src/vs/workbench/contrib/mcp/common/

Bridge: McpLanguageModelToolContribution (mcpLanguageModelToolContribution.ts:41) watches McpService.servers and auto-registers each server's tools into ILanguageModelToolsService.

Discovery sources:

  • VS Code settings (mcp.servers in settings.json)
  • Claude Desktop config (claude_desktop_config.json) — adapter at nativeMcpDiscoveryAdapters.ts:64
  • Windsurf (~/.codeium/windsurf/mcp_config.json)
  • Cursor (~/.cursor/mcp.json)
  • Workspace .vscode/mcp.json
  • Extension contributions

For our use case: VS Code hosts MCP servers centrally. Our agent queries getTools(), serializes tool definitions to the CLI assistant, and routes tool calls back through invokeTool().

8.3 Skills & Custom Agents

Skills are .prompt.md files discovered by IPromptsService. They're prompt injections, not callable tools.

Custom agents (ICustomAgent at promptsService.ts:225) are .prompt.md files with agent frontmatter, offering tools, model, agents (sub-agents), hooks, and agentInstructions fields.

Skills are filtered by sessionType, so they can be scoped per CLI assistant.

12. Virtualization & Performance

List virtualization: ChatListWidget wraps WorkbenchObjectTree (backed by VS Code's standard ListView). Items scrolling out of view trigger disposeElement() which clears disposables and codeblock URI mappings.

Editor pooling: EditorPool (chatContentCodePools.ts:15) is a KeyedResourcePool with maxIdleSize: 2:

  • Code blocks acquire editors from the pool on render, release on scroll-out
  • Keyed reuse — tries to return the same editor for the same code block when scrolling back
  • 10-second debounce before trimming excess idle editors (prevents churn during scrolling/streaming)
  • Separate DiffEditorPool for diff views

Cost model: N editors for visible code blocks + 2 warm spares. Monaco construction cost is amortized through pooling.

13. Reusing Components Outside the Chat Panel

The ChatWidget is designed to be embedded in different containers.

Embedding a read-only chat widget:

const widget = instantiationService.createInstance(ChatWidget,
    location, viewContext,
    {
      rendererOptions: { noHeader: true, noFooter: true, editable: false },
      renderFollowups: false,
      renderStyle: 'compact',
    },
    styles

// Load historical data
const modelRef = chatService.loadSessionFromData(serializedData);
widget.setModel(modelRef.object);

// Or add individual complete request/response pairs
chatService.addCompleteRequest(sessionResource, 'user msg', variableData, 0, {
  message: new MarkdownString('**response**'),
});

Standalone markdown rendering (without chat session machinery):

const renderer = instantiationService.createInstance(ChatContentMarkdownRenderer);
const result = renderer.render(new MarkdownString('**bold** and `code`'));
parentElement.appendChild(result.element);

14. Shared Lib Feasibility (Nx Monorepo / Web App)

Direct component extraction is not feasible. The components are deeply coupled to VS Code's DI system (14+ injected services per content part). But a shared data layer is practical.

Why direct extraction fails

  • ChatMarkdownContentPart alone injects 14 services: IContextKeyService, IConfigurationService, IEditorService, IModelService, ILanguageService, IMenuService, IHoverService, IChatService, etc.
  • CodeBlockPart creates full CodeEditorWidget / DiffEditorWidget instances
  • The DI system (createDecorator / @IService) requires the full InstantiationService bootstrap
  • Chat components are NOT part of Monaco editor — they live in the workbench layer above it

Recommended Nx structure

nx-monorepo/
  libs/
    chat-types/          ← SHARED: IChatProgress, ISerializableChatData,
                            IExportableChatData (pure TS, zero deps)
    chat-markdown/       ← SHARED: base markdown renderer extracted from
                            vs/base/browser/markdownRenderer.ts + marked parser
    chat-css/            ← SHARED: CSS custom properties, shimmer animation,
                            layout vars (remap --vscode-* tokens)

  apps/
    code-oss/            ← VS Code fork (native components as-is)
    web-app/             ← Web app with own components consuming shared types
      chat-renderer/     ← React/Svelte components using chat-types,
                            chat-markdown, Monaco standalone for code blocks

15. Multi-Window Routing (Remote Web Server)

Architecture: one extension server, many browser tabs

┌───────────────────────────────────────────────────────┐ │ Server (one per machine) │ │ │ │ ┌───────────────────────────────────────────┐ │ │ │ Extension Host (Remote) │ │ │ │ ExtHostChatAgents2 (ONE instance) │ │ │ │ - our extension runs HERE │ │ │ │ - handles $invokeAgent(requestId, ...) │ │ │ │ - sends $handleProgressChunk(requestId, ..) │ │ │ └───────────────┬───────────────────────────┘ │ │ │ RPC per connection │ └────────────────┬──────────────────────────────────────┘ ┌─────────┴─────────┐ │ │ ┌────┴────┐ ┌────┴────┐ │ Browser │ │ Browser │ ← one per worktree │ Tab 1 │ │ Tab 2 │ │ │ │ │ │ MainThreadChatAgents2 │ ← one per browser tab │ ChatServiceImpl │ ← one per browser tab │ ChatWidget │ ← one per browser tab │ _pendingProgress map │ ← scoped per tab └─────────┘ └─────────┘

Routing mechanism: requestId

  1. Browser tab 1's ChatWidget._acceptInput() calls chatService.sendRequest()
  2. ChatServiceImpl creates a unique requestId, stores the progressCallback in MainThreadChatAgents2._pendingProgress keyed by requestId
  3. Request sent to extension host via RPC: $invokeAgent(handle, request)
  4. Extension processes request, calls $handleProgressChunk(requestId, chunks)
  5. RPC layer routes the response to the correct browser tab's MainThreadChatAgents2 — the proxy is scoped per connection
  6. That tab's _pendingProgress.get(requestId) finds the right callback
  7. Progress flows to the right ChatModel → right ChatWidget

No session ID routing needed. Each browser tab has its own service instances. The RPC channel is per-connection. Our extension doesn't need to know which UI it's talking to — it just responds to the request it received.

One thing to track: If our extension spawns a long-running CLI process per chat session, we need to map requestId / sessionResource to the right process for cancellation.

16. Recommended Architecture

Phase 1: Core Pipeline

  1. Embed code-oss served as a web app (one instance per worktree, as already done)
  2. Build an extension that registers a chat participant via vscode.chat.createChatParticipant(id, handler) with isDefault: true
  3. In the handler:
    • Receive (request, context, response, token)
    • Spawn / connect to the selected CLI assistant as a child process
    • Parse CLI stdout and map to response.markdown(), .thinkingProgress(), .beginToolInvocation(), etc.

Phase 2: UI Customization

  • Replace model picker with assistant picker (installed vs. available, with Install button)
  • Scope capabilities per assistant via IChatAgentData.capabilities
  • Scope skills/commands per assistant via sessionType filtering
  • MCP: VS Code hosts MCP servers centrally; agent serializes tool definitions to CLIs and routes tool calls back through invokeTool()

Phase 3: Web App Integration

  • Share chat-types (pure TS interfaces) in Nx monorepo lib
  • Share CSS design tokens (remap --vscode-* variables)
  • Web app renders with own components + Monaco standalone for code blocks
  • Use ISerializableChatData as the interchange format
