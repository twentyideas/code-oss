DevSwarm Chat Architecture v2

Replacing the Copilot agent pipeline with CLI-driven coding assistants

May 2026 · DRAFT — investigation ongoing

1. Core Concept

The user selects a coding assistant (Claude Code, Codex, Copilot CLI, etc.), not a model or agent. That selection determines which CLI process drives the entire chat session. VS Code's chat UI renders the output. Everything in between is translation.

3. The Participant Question

3.1 What Copilot's Participants Actually Are

Copilot registers multiple chat participants in VS Code, each mapped to an "Intent":

ParticipantIntentWhat It Does
github.copilot.defaultUnknown / AskAgentDefault handler — routes to Copilot's LLM
@workspace(separate extension)Workspace-aware context — searches codebase, reads files
@terminalTerminalTerminal-aware — suggests commands, explains output
@vscodeVSCodeVS Code settings/API expert
Editing agentEditMulti-file editing with diff view
Edits agentAgent (agentic)Full autonomous agent with tool use
Notebook agentsEditor / notebookEditorNotebook-specific editing

These are Copilot-specific routing categories. Each one calls the same Copilot LLM backend with different system prompts and context strategies. They're how Copilot decides whether to search your codebase, suggest a terminal command, or edit files.

3.2 Do CLI Assistants Have Equivalents?

No. CLI coding assistants don't have "participants." They're single-process agents that handle everything internally:

  • Claude Code — One process. It decides when to read files, run commands, edit code. No @workspace/@terminal split. Has its own tool system (Read, Edit, Bash, etc.), its own MCP client, its own permission model.
  • Codex CLI — One process. Has its own tool execution, sandboxing, approval flow.
  • Copilot CLI (gh copilot) — Simple explain/suggest. No agentic behavior, no @-mentions.

The CLI IS the @workspace + @terminal + @vscode + editing agent rolled into one. It doesn't need routing because there's nothing to route between.

3.3 What This Means for Us

We do not need to replicate the participant system. We don't need to forward participant info to the CLI. When the user types a message, it goes straight to the CLI process. The CLI decides internally what to do — read files, run terminal commands, edit code.

The only thing we might lose is the ability to type @workspace or @terminal to explicitly scope a query. But CLI assistants handle this via natural language ("search my codebase for X", "run this command") rather than @-mention routing.

5. What Happens to sendRequest()

5.1 Current _sendRequestAsync() (simplified)

private _sendRequestAsync(model, sessionResource, parsedRequest, ...) {
  const progressCallback = (progress: IChatProgress[]) => {
    model.acceptResponseProgress(request, progressItem);   // ← KEEP THIS
  };

  // --- STRIP ALL OF THIS ---
  const agentPart = parsedRequest.parts.find(...);           // agent detection
  const commandPart = parsedRequest.parts.find(...);         // command detection
  await this.chatAgentService.detectAgentOrCommand(...);     // participant detection
  await this.extensionService.activateByEvent(...);          // agent activation
  const history = this.getHistoryEntriesFromModel(...);      // history for agent
  const agentResult = await this.chatAgentService.invokeAgent(  // THE COPILOT CALL
    agent.id, requestProps, progressCallback, history, token
  );
  // --- END STRIP ---

  // --- REPLACE WITH ---
  const agentResult = await this.devswarmService.sendToAssistant(
    selectedAssistantId, requestProps, progressCallback, token
  );
}

5.2 The RPC Boundary

Why we can't skip the RPC hop: sendRequest() runs in the browser. The PTY runs on the server. The extension host / DevSwarm ExtHost service is the bridge. We need a MainThread/ExtHost service pair, but it's far simpler than the agent protocol:

  • MainThread → ExtHost: $sendToAssistant(assistantId, message, requestId, context)
  • ExtHost → MainThread: $handleProgress(requestId, IChatProgress[])
  • MainThread → ExtHost: $cancelRequest(requestId)

That's the entire protocol. Three methods. No agent resolution, no participant detection, no model lookup.

Each result is tagged with a PromptFileSource (e.g., ClaudeWorkspace, CopilotPersonal, GitHubWorkspace).

7.2 What's Missing

Problem: The discovery merges everything together. There's no concept of "show only Claude's skills when Claude Code is selected." The PromptFileSource tags exist but nothing filters by active assistant.

Solution: Add filtering in the completion providers and skill lists. When the assistant picker says "Claude Code," filter discovery results to ClaudeWorkspace / ClaudePersonal sources. When it says "Codex," filter to Codex sources (after adding Codex paths to the discovery arrays).

For CLIs with non-.prompt.md formats: Build adapters that read the CLI's native config and present commands as discoverable items.

7.3 Slash Commands vs. CLI Commands

Important distinction: VS Code's slash commands (/fix, /explain, /tests) are Copilot's participant subcommands. They route to different intents within Copilot's LLM call. They are NOT the CLI's own commands.

Claude Code has its own /commands (from .claude/commands/). Codex may have its own. These are different systems.

Options:

  • Query each CLI for its commands at startup and register them dynamically
  • Read the CLI's command directory and present as slash commands
  • Forward the raw /command text to the CLI and let it handle parsing

8. Tool/MCP: Two Separate Worlds

VS Code's World CLI Assistant's World ┌───────────────────────────────┘ ┌────────────────────────────────┘ │ ILanguageModelToolsService │ │ Claude Code's internal tools │ │ - EditTool, ReadTool, etc. │ │ - Read, Edit, Bash, Search, etc. │ │ - vscode.lm.tools API │ │ - its own tool execution engine │ ┐───────────────────────────────┌ ┐────────────────────────────────┌ ┌───────────────────────────────┘ ┌────────────────────────────────┘ │ MCP servers via settings.json │ │ Claude's MCP via │ │ - auto-discovered from Claude │ │ ~/.claude/settings.json │ │ Desktop, Cursor, Windsurf │ │ - its own MCP client │ ┐───────────────────────────────┌ ┐────────────────────────────────┌ ┌───────────────────────────────┘ ┌────────────────────────────────┘ │ .prompt.md skills │ │ Claude's /commands from │ │ - .github/skills/ │ │ .claude/commands/ │ │ - .claude/skills/ │ │ - its own slash command system │ ┐───────────────────────────────┌ ┐────────────────────────────────┌ These are PARALLEL systems. They don't talk to each other. The CLI manages its own tools/MCP/commands internally. VS Code's system was built for Copilot's architecture where VS Code is the agent runtime.

Our role: We're not bridging these systems. The CLI runs independently. What we CAN do:

  • Surface the CLI's tool activity in the UI via beginToolInvocation progress chunks (read-only visibility)
  • Share MCP configs between VS Code and the CLI (read the CLI's config, or write VS Code's MCP to the CLI's config)
  • Surface the CLI's commands as slash commands in the input (discovery bridge)

10. Implementation Phases

Phase 1: Core Pipeline (MVP)

  1. Create DevSwarmChatViewPane — our own view pane wrapping ChatWidget
  2. Hide native ChatViewPane at source level (change ctorDescriptor in chatParticipant.contribution.ts:70)
  3. Strip mode/model pickers from the input toolbar
  4. Add assistant picker (extension contribution point)
  5. Create DevSwarm MainThread/ExtHost service pair (3 methods: send, progress, cancel)
  6. Simplify _sendRequestAsync() to use DevSwarm service instead of agent pipeline
  7. Wire PTY manager output → IChatProgress translation → progressCallback

Phase 2: Discovery & Commands

  1. Add filtering to prompt file discovery by active assistant (use PromptFileSource tags)
  2. Bridge CLI native commands to slash command completions
  3. Surface CLI tool activity as beginToolInvocation progress in the chat UI
  4. Handle context attachments — pass @file contents to CLI

Phase 3: MCP & Tool Management

  1. Build per-assistant MCP config management UI
  2. Optional: share MCP configs between VS Code and CLI assistants
  3. Surface assistant-specific settings/permissions in the chat
  4. Handle CLI approval/permission prompts in the VS Code UI

Phase 4: DevSwarm Extension API (vscode.devswarm)

  1. Formalize the assistant lifecycle hooks as a public API namespace
  2. devswarm.assistants — register providers, query installed, trigger install
  3. devswarm.terminals — provision CLI terminals, stream I/O
  4. Our extension becomes the first consumer, but other extensions can use the same hooks
